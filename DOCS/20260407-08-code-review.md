## Critical Issues

### 1. `parseFloat(price) || 0` breaks on price = `"0"` — `utils/schema.ts:100`

```ts
"price": parseFloat(o.price) || 0,
```

`parseFloat("0")` returns `0`, which is falsy, so `|| 0` is a no-op but the real problem is if a free workshop has a price of `"0"` — it will correctly produce `0`. However, the `|| 0` pattern silently swallows `NaN` instead of flagging it. A price of `"sur devis"` still produces `0` in the schema with no warning to the editor.

**Fix:** Validate explicitly so bad data surfaces:

```ts
const numericPrice = parseFloat(o.price);
"price": Number.isFinite(numericPrice) ? numericPrice : 0,
```

Better still, log a console warning in dev when price is non-numeric.

### 2. Markdoc HTML rendered without sanitisation — `blog/[slug].astro:24`

```ts
const renderedContent = transformed ? Markdoc.renderers.html(transformed) : '';
// ...
<Fragment set:html={renderedContent} />
```

`Markdoc.renderers.html()` produces a raw HTML string that is then injected with `set:html`. Astro does **not** sanitise `set:html`. If any Markdoc content ever contains a raw HTML node (allowed by default Markdoc config), an editor could inject arbitrary HTML including `<script>` tags.

**Fix:** Either disable Markdoc's `html` node type in the transformer config, or pass the output through a sanitiser (e.g., `sanitize-html` or `DOMPurify`). For a small editorial site with trusted authors this is low practical risk, but worth documenting explicitly.

### 3. Gallery button missing `type="button"` — `mes-creations.astro:92`

```astro
<button class="gallery-item w-full" data-src={item.src.src} data-alt={item.alt}>
```

A `<button>` without `type` defaults to `type="submit"`. If this ever ends up inside a `<form>`, clicking a gallery image will submit the form. Add `type="button"`.

## Architectural & Design Concerns

### 4. Gallery images resolved twice — `mes-creations.astro:24–31` and `55–60`

`galleryItems` is built with `Promise.all` (lines 24–31), but the result contains `{ src: ImageMetadata | null, ... }`. The schema block then re-filters with a type predicate (line 55) to remove nulls. This is correct but the null items are still being computed and allocated. The resolution could short-circuit at collection time:

```ts
// Resolve and discard nulls in one pass
const galleryItems = (await Promise.all(
  sorted.map(async (creation) => {
    const src = await resolveImage(creation.entry.image, creationGlob);
    if (!src) return null;
    return { src, alt: creation.entry.imageAlt || creation.entry.title, title: creation.entry.title };
  })
)).filter((item): item is NonNullable<typeof item> => item !== null);
```

This removes the need for the second filter and the `!` assertion in the schema block entirely.

### 5. `ATELIER_GROUP_LABELS` cast is unnecessary — `utils/ateliers.ts:13–15`

```ts
export const ATELIER_GROUP_LABELS = Object.fromEntries(
  ATELIER_GROUPS.map((g) => [g.id, g.label]),
) as Record<AtelierGroupId, string>;
```

`Object.fromEntries()` widens to `Record<string, string>` so the cast is needed — but it hides the problem. Since `AtelierGroupId` is a union of string literals, TypeScript can't prove at compile time that `fromEntries` produces exactly that shape. A safer alternative uses `reduce` or a typed helper:

```ts
export const ATELIER_GROUP_LABELS = ATELIER_GROUPS.reduce(
  (acc, g) => { acc[g.id] = g.label; return acc; },
  {} as Record<AtelierGroupId, string>,
);
```

This is equivalent but makes the intent explicit without relying on a cast.

### 6. `ATELIER_GROUP_LABELS` lookup uses `as keyof` cast — `ateliers-reguliers.astro:115`

```astro
{ATELIER_GROUP_LABELS[c.group as keyof typeof ATELIER_GROUP_LABELS] || c.location}
```

Keystatic's select field guarantees `c.group` is one of the option values — but TypeScript infers it as `string` because keystatic's generated types don't narrow to a union. The cast is the only practical option here, but it means TypeScript won't warn if a new group is added to `ATELIER_GROUPS` but not to keystatic's select options (or vice versa). The existing check in keystatic config now imports `ATELIER_GROUPS` so the options stay in sync — this is good. The cast is acceptable; add a comment explaining why.

### 7. Lightbox has no focus management — `mes-creations.astro:140–163`

When the `<dialog>` opens via `showModal()`, browsers should move focus to the first focusable element inside it — but the close `<button>` is the only focusable element and it has `position: absolute`, which some browsers don't include in the tab sequence by default. The `<img>` has no `tabindex`. When the dialog closes, focus is not explicitly restored to the triggering button.

**Fix:** Capture the triggering button reference and restore focus on close:

```ts
let lastFocused: HTMLElement | null = null;

document.querySelectorAll('.gallery-item').forEach((item) => {
  item.addEventListener('click', () => {
    lastFocused = item as HTMLElement;
    // ...
    dialog.showModal();
    dialog.querySelector('button')?.focus();
  });
});

dialog?.addEventListener('close', () => lastFocused?.focus());
```

## Code Quality

### 9. `formatFrenchDate` does not validate input — `utils/strings.ts:9–15`

```ts
export function formatFrenchDate(dateString: string): string {
  return new Date(dateString).toLocaleDateString('fr-FR', { ... });
}
```

`new Date("invalid")` returns an Invalid Date object, and `toLocaleDateString()` on it returns the string `"Invalid Date"`, which would render visibly on the page. Called from `blog/[slug].astro:30` with a post's `publishDate` which is always a valid date string from Keystatic — but the function's contract doesn't enforce this.

**Fix:**

```ts
export function formatFrenchDate(dateString: string): string {
  const date = new Date(dateString);
  if (isNaN(date.getTime())) return '';
  return date.toLocaleDateString('fr-FR', { year: 'numeric', month: 'long', day: 'numeric' });
}
```

### 10. `blog/[slug].astro` calls `formatFrenchDate` without null guard — line 30

```ts
const formattedDate = post.publishDate ? formatFrenchDate(post.publishDate) : '';
```

This is correct — the ternary guards the call. But `post.publishDate` is typed as `string | null` by Keystatic, yet articles can only appear in `getStaticPaths` if they have a `publishDate` (line 13 filters for it). The `post.publishDate ? ... : ''` check is defensive dead code at runtime, though TypeScript requires it for type correctness. A comment would clarify this invariant.

### 11. `buildServicePageSchemas` mutates the result of `buildPageSchemas` — `utils/schema.ts:87–103`

```ts
const schemas = buildPageSchemas(...);  // returns a new array
schemas.push({ "@type": "Service", ... });  // mutates it
if (faqItems.length > 0) schemas.push({ "@type": "FAQPage", ... });
return schemas;
```

`buildPageSchemas` returns a fresh array, so mutation is safe at runtime — but it's surprising. Callers that spread the result (like `mes-creations.astro:45`) would see stale data if they cached the schemas reference. This is not a bug today but is a subtle contract violation.

**Fix:** Build the full array declaratively:

```ts
return [
  ...buildPageSchemas({ title, seoDescription, pageUrl, siteUrl, settings }),
  { "@type": "Service", ... },
  ...(faqItems.length > 0 ? [{ "@type": "FAQPage", ... }] : []),
];
```

### 12. `siteUrl` used in schema without null guard — `blog/index.astro:17`, `blog/[slug].astro:25`

```ts
const siteUrl = Astro.site?.origin;  // string | undefined
// later...
"item": siteUrl,  // could be undefined
```

`Astro.site` is configured in `astro.config.mjs` for production but may not be set in some dev configurations. Using `undefined` as a schema URL produces invalid JSON-LD. All service pages use `Astro.site?.origin ?? ''` to handle this — the blog pages should do the same.

### 13. Cover glob re-created on every page — all seven pages

Every service page and blog page creates:

```ts
const coverGlob = import.meta.glob<{ default: ImageMetadata }>('/src/assets/images/covers/**/*.{jpg,jpeg,png,webp}');
```

`import.meta.glob` is resolved at build time by Vite, so there's no runtime performance cost. However, the type annotation is repeated seven times. If the glob pattern ever changes (e.g., adding `.avif`), all seven files need updating.

**Fix:** Export a shared glob from `utils/images.ts`:

```ts
// This must stay in utils/images.ts — import.meta.glob only works statically
export const coverGlob = import.meta.glob<{ default: ImageMetadata }>('/src/assets/images/covers/**/*.{jpg,jpeg,png,webp}');
```

Actually, `import.meta.glob` can only be called statically in the file where it's written — it cannot be moved to a utility module in Vite/Astro. So this cannot be extracted. Add a comment on each page explaining why it's repeated to prevent future "DRY violation" confusion.

### 16. `!important` utilities in ContactCTA are a code smell — `ContactCTA.astro:19`

```astro
class="... !text-white ... !no-underline"
```

The `!` prefix in Tailwind generates `!important` CSS. This was added to beat `.prose a` specificity. The correct fix is to ensure ContactCTA is never a direct descendant of a `.prose` container without a `.not-prose` wrapper. In `la-couturiere.astro` the wrapping div already has `class="text-center not-prose"` — so the `!important` is already unnecessary there. Check all other usages; if `.not-prose` is always present, remove the `!` prefixes.

## Accessibility

### 18. Lightbox image has no `alt` text when opened — `mes-creations.astro:140–152`

```ts
lightboxImg.src = el.dataset.src;
lightboxImg.alt = el.dataset.alt || '';
```

`el.dataset.alt` comes from `data-alt={item.alt}` on the button. If `item.alt` is the creation title (which it is), this should be populated — but the fallback is `''`, which is the correct value for a decorative image. The `<img>` in the dialog starts with `alt=""` on render, which is correct until JS runs. This is fine.


No `focus-visible` ring is defined on `.gallery-item`. Keyboard users tabbing through the gallery have no visual indication of which image is focused. Add `focus-visible:ring-2 focus-visible:ring-[var(--color-button)]`.

- **`filterPublishedPosts` type predicate** is clean and correctly narrows the type. The generic constraint `T extends { entry: { publishDate?: string | null } }` is well-scoped.
- **`ATELIER_GROUPS as const`** with a derived `AtelierGroupId` union is idiomatic TypeScript.
- **`buildPageSchemas` / `buildServicePageSchemas` hierarchy** in `schema.ts` is the right composition pattern. The JSDoc comments are helpful.
