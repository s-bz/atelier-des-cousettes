## Critical Issues

### 1. Non-null assertion after filter does not narrow the type — `mes-creations.astro:58`

```astro
.filter(item => item.src)
.map(item => ({
  "contentUrl": new URL(item.src!.src, Astro.site).toString(),  // ← unsafe
```

`.filter(item => item.src)` does not narrow `item.src` to non-null in TypeScript; the type of `item.src` remains `ImageMetadata | null` inside the `.map`. The non-null assertion (`!`) silences the compiler but will throw at runtime if a null somehow passes through (e.g., if the filter predicate changes).

**Fix:** Use a type predicate filter:

```ts
.filter((item): item is typeof item & { src: ImageMetadata } => item.src !== null)
.map(item => ({
  "contentUrl": new URL(item.src.src, Astro.site).toString(),
```

---

### 2. `Number(price)` silently produces `NaN` for non-numeric strings — `utils/schema.ts:100`

```ts
"price": Number(o.price),
```

If a CMS editor enters `"40-90"`, `"sur devis"`, or `"40 €"`, `Number()` returns `NaN`. Schema.org validators will reject this and Google Search Console will flag it.

**Fix:** Either parse only the first numeric portion, or validate the field in keystatic config with a pattern hint:

```ts
"price": parseFloat(o.price) || 0,
```

And update the Keystatic field label: `'Prix (nombre seul, ex: 40)'` to set expectations.

## Architectural & Design Concerns

### 3. FAQ section duplicated across four page files

The FAQ rendering block is copy-pasted identically in `stages-thematiques.astro`, `ateliers-reguliers.astro`, `un-apres-midi-couture.astro`, and `la-couturiere.astro`:

```astro
<section class="bg-[var(--color-bg-accent)] py-16 md:py-20">
  <div class="max-w-3xl mx-auto px-4">
    <h2 ...>{page.faqSectionTitle || 'Questions fréquentes'}</h2>
    <div class="space-y-6">
      {faqItems.map((item) => (...))}
    </div>
    <ContactCTA ... />
  </div>
</section>
```

Any styling or structural change must be applied in four places. Extract a `<FaqSection>` component:

```astro
---
// src/components/FaqSection.astro
interface Props { title?: string; items: { question: string; answer: string }[]; ctaLabel?: string; }
---
```

### 4. Service card components (`AtelierCard`, `StageCard`) are near-identical

Both components share the same card shell, shadow, hover pattern, accent badge, and `<a>` link structure. The only meaningful differences are the displayed fields (day/time vs price/duration).

Consolidate into a single `<ServiceCard>` component with configurable slots or a `badges` prop array, reducing duplication and ensuring visual consistency.

### 5. Atelier group definitions exist in two places

`keystatic.config.ts` lines 172–178 define group option values (`revel-adultes`, `revel-enfants`, `verdalle`). `ateliers-reguliers.astro` lines 34–38 re-define the same groups as a typed `const` array for rendering. These are two sources of truth — if a group is added in Keystatic but not in the page file (or vice versa), the grouping silently breaks.

**Fix:** Export the group definitions from a shared module (e.g., `src/utils/ateliers.ts`) and import in both the config and the page.

### 6. `blog/[slug].astro` reads `blogIndex` singleton on every article render

```ts
const blogIndex = await reader.singletons.blogIndex.read();
```

This is a singleton read on every article page request. It's not expensive, but it's semantically odd — the CTA text for articles is borrowed from the listing page. Consider adding a dedicated `articleCtaText` / `articleCtaLabel` to `blogIndex` to make the intent explicit, or accept the current coupling and add a comment explaining why.

## Code Quality

### 7. Date formatting duplicated in `BlogCard.astro` and `blog/[slug].astro`

`BlogCard.astro:11–15` and `blog/[slug].astro:29–34` both call `toLocaleDateString('fr-FR', { year: 'numeric', month: 'long', day: 'numeric' })` with identical options. Extract to `src/utils/strings.ts`:

```ts
export function formatFrenchDate(dateString: string): string {
  return new Date(dateString).toLocaleDateString('fr-FR', {
    year: 'numeric', month: 'long', day: 'numeric',
  });
}
```

### 8. Published post filter logic duplicated in both blog files

`blog/index.astro:16–18` and `blog/[slug].astro:12–13` both filter by `publishDate <= now`. Extract to `src/utils/reader.ts` or a `src/utils/blog.ts` helper:

```ts
export function filterPublishedPosts<T extends { entry: { publishDate?: string | null } }>(posts: T[]): T[] {
  const now = new Date();
  return posts.filter(p => Boolean(p.entry.publishDate) && new Date(p.entry.publishDate!) <= now);
}
```

### 9. `.split()` called without null/empty guard in multiple pages

The following assume the field is always a non-empty string, which is only the case if the CMS editor fills it in:

- `ateliers-reguliers.astro`: `c.priceDetails.split('\n\n')`, `c.dates.split('\n')`
- `stages-thematiques.astro`: `stage.fullDescription.split('\n\n')`, `stage.dates.split('\n')`
- `un-apres-midi-couture.astro`: `page.description.split('\n\n')`, `page.dates.split('\n')`
- `la-couturiere.astro`: `section.text.split('\n\n')`

If any optional field is left blank, `.split()` on an empty string returns `['']`, rendering an empty `<p>` tag. Guard with a truthiness check:

```astro
{stage.dates && stage.dates.split('\n').map((line) => <p>{line}</p>)}
```

### 10. `settings?.authorName` renders `"Par undefined"` without fallback — `blog/[slug].astro:89`

```astro
<span>Par {settings?.authorName}</span>
```

If `siteSettings` is missing or `authorName` is not set, this renders `"Par undefined"` visibly on the page. Add a fallback:

```astro
<span>Par {settings?.authorName ?? 'L\'Atelier des Cousettes'}</span>
```

Or conditionally hide the author metadata row if settings are absent.

### 11. `Hero.astro` has a hardcoded magic colour — `Hero.astro:29`

```astro
class="... bg-[#2c2420] ..."
```

This bypasses the CSS custom properties system documented in CLAUDE.md. Should be `bg-[var(--color-bg-dark)]` or equivalent token, so it participates in any future theme changes.

### 12. `ContactCTA.astro` default label is hardcoded French — `ContactCTA.astro:10`

```ts
label = "Demande d'information"
```

This is the single most-rendered user-facing string in the site yet it lives in a component default, not in Keystatic. Add a `defaultCtaLabel` field to `siteSettings` so editors can update it without a code deploy.

### 13. Hardcoded cross-link anchor text in multiple pages

The text "Ateliers réguliers →", "Stages thématiques →", "Un après-midi couture →" appears hardcoded in `mes-creations.astro:123–131`, `stages-thematiques.astro:137–142`, `ateliers-reguliers.astro:172–177`, and `un-apres-midi-couture.astro:162–167`. Per CLAUDE.md, UI text should not be hardcoded. These strings are unlikely to change, but if the page titles ever do, the links will be stale.

At minimum, define them as shared constants in a `src/utils/nav.ts` module.

### 14. Hardcoded empty-state strings in `mes-creations.astro`

```astro

<p class="...">Galerie en cours de construction.</p>  {/* line 112 */}
```

## Type Safety

### 15. Keystatic `price` fields typed as `text` with no format enforcement

`schemaOffers`, `price` (stage), `price` (après-midi couture) are all free-text fields. `schema.ts` casts them with `Number()` which produces `NaN` silently (see Critical Issue #2). Consider using `fields.number()` in Keystatic for price fields that feed into schema markup, or use `fields.text()` with a dedicated validator.

### 16. `la-couturiere.astro` section accent logic is fragile — line 39

```ts
const lastSectionHasAccent = (resolvedSections.length - 1) % 2 === 0;
```

This hardcodes the assumption that sections alternate background every index. If CMS structure changes (e.g., sections get a dedicated `hasAccentBackground` field), this arithmetic becomes wrong and the cross-links section will have the wrong background with no error. The logic should either be made explicit (field per section) or documented with a comment explaining the invariant.
