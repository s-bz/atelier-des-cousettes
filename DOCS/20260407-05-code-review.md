## Architectural & Design Concerns

### 1. `la-couturiere.astro` builds schemas inline while other service pages use `buildServicePageSchemas()`

**File:** `src/pages/la-couturiere.astro:32-72`

The three service pages now share `buildServicePageSchemas()` from `src/utils/schema.ts` — a good DRY improvement. However, `la-couturiere.astro` still constructs its JSON-LD schemas inline (BreadcrumbList + WebPage + Person). The Person schema is unique to this page, but the BreadcrumbList and WebPage blocks are identical to the shared utility's output. Consider either:
- Extracting a lighter `buildPageSchemas()` helper that handles BreadcrumbList + WebPage (without Service/FAQ), then appending the Person schema in the page.
- Or accepting the duplication since this is the only non-service page with custom schema needs.

**Severity:** Low — one instance of duplication, but worth noting for future pages.

### 2. Hardcoded cross-links heading in Astro file

**File:** `src/pages/la-couturiere.astro:136`

The heading `"Découvrez mes formules de cours"` is hardcoded in the `.astro` template. Per CLAUDE.md, all user-facing content should be managed through Keystatic. Consider adding a `crossLinksSectionTitle` field to the Keystatic schema (as the other pages have for their section titles).

### 3. Orphaned `public/images/couturiere/` directory

The old Markdoc content referenced images from `public/images/couturiere/`. The new YAML content correctly references `src/assets/images/couturiere/` (Astro-optimized). However, the `public/images/couturiere/` directory still exists with 3 unused files (~345kB). These should be deleted to avoid confusion and reduce deploy size.

## Code Quality Improvements

### 4. Schema utility is a clean DRY win

**File:** `src/utils/schema.ts`

The new `buildServicePageSchemas()` utility is well-typed with clear interfaces and eliminates ~35 lines of duplicated schema construction from each service page. Good use of a shared abstraction for a genuine pattern (3+ consumers). The `readonly` arrays in the interface are a nice touch.

### 5. Missing `dateModified` in `la-couturiere.astro` WebPage schema but present in shared utility

**File:** `src/pages/la-couturiere.astro:47` vs `src/utils/schema.ts:53-67`

The la-couturière page includes `"dateModified"` in its inline WebPage schema, but the shared `buildServicePageSchemas()` utility omits it. This is an inconsistency — either both should include it or neither should. The `dateModified` with `new Date()` is of questionable SEO value anyway (it reports today's date on every build, not the actual content modification date).

### 6. Cross-links section background alternation logic is fragile

**File:** `src/pages/la-couturiere.astro:133`

```typescript
class={resolvedSections.length % 2 === 0 ? 'bg-[var(--color-bg-accent)]...' : '...'}
```

This depends on the exact number of sections to alternate backgrounds correctly. If someone adds or removes a section in Keystatic, the cross-links section may end up with the same background as the section above it. The other pages don't have this issue because their section order is fixed in the template. Consider computing the background based on the last rendered section rather than total count.
