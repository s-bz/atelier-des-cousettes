## Architectural & Design Concerns

### 1. `location` field exists in Keystatic schema but is never rendered

`keystatic.config.ts:216` defines a `location` field and `index.yaml:13-15` populates it with a useful string ("Maison des associations de Revel et atelier privé de Verdalle (Tarn). Le lieu est précisé lors de l'inscription."). However, `un-apres-midi-couture.astro` never reads or renders `page.location`.

Visitors are left without location information on the page itself — they can only find it buried in the FAQ answer ("À la maison des associations de Revel…"). Since the FAQ answer on line 52 in the YAML essentially duplicates the `location` field, the location is stored twice with no single authoritative source.

**Fix:** Either render `page.location` as a third info badge (after price and duration) in the intro section, or remove the field from Keystatic and keep only the FAQ answer. Displaying it in the intro is the better choice for UX.

### 2. `price` and `duration` as top-level flat fields are partially redundant with `schemaOffers`

The page has both `price: '40 €'` / `duration: '3h'` flat fields (for display) and `schemaOffers[0].price: '40'` (for schema markup). The price value is thus stored twice — if Isabelle updates the price in the CMS, she must update it in two places.

On `stages-thematiques` and `ateliers-reguliers`, price is stored only in the structured `stages[]` / `creneaux[]` arrays and `schemaOffers[]` is a separate SEO-only concern. This page breaks that convention.

**Option:** Keep the flat fields for display (they're simpler to edit for a single-offering page) but document in the Keystatic label that `schemaOffers` must be updated in sync. Or derive the display from `schemaOffers[0]` directly, removing the flat fields.

### 3. JSON-LD / FAQ schema block still duplicated (known carry-over)

The BreadcrumbList + WebPage + Service + FAQPage construction in `un-apres-midi-couture.astro:46-82` is the fourth verbatim copy of this pattern across service pages. Noted in review 03 (#2). This page adds no new variations — it is a direct copy. A `src/utils/schema.ts` helper would now save ~40 lines across four files.

## Code Quality Improvements

### 4. `audienceItems[]` uses bare text — inconsistent with other array fields

In Keystatic, `audienceItems` is defined as `fields.array(fields.text(...))` — an array of plain strings. All other array fields in the project use `fields.array(fields.object(...))` with named properties. While appropriate for a simple list, it means Keystatic editor UX differs from the rest of the config (no object wrapper, no `itemLabel` that reads a named field).

This is a minor inconsistency, not a bug. The `itemLabel: (props) => props.value` pattern is correct for the text field variant. No change required unless you want to add a `label` property to each item in the future (e.g., for icons or styling variants).

### 5. Hardcoded UI strings remain

Consistent with the pattern flagged in review 03 (#5), several strings are hardcoded in the template:

The `faqSectionTitle` fallback `'Questions fréquentes'` is consistent with `stages-thematiques.astro` and `ateliers-reguliers.astro`. The badge sub-labels ("la séance", "d'accompagnement") are UI chrome tightly coupled to the price/duration fields — they would be reasonable to leave hardcoded, or move to Keystatic if Isabelle ever wants different labels (e.g., "par séance").
