## Architectural & Design Concerns

### 1. `toSlug()` is duplicated across two page files

`stages-thematiques.astro:43` and `ateliers-reguliers.astro:54` define the same helper verbatim. When `un-apres-midi-couture` is redesigned it will be copy-pasted a third time.

**Fix:** Move to `src/utils/strings.ts` and import in both pages:

```ts
// src/utils/strings.ts
export const toSlug = (name: string) =>
  name.toLowerCase().normalize('NFD')
    .replace(/[\u0300-\u036f]/g, '')
    .replace(/[^a-z0-9]+/g, '-')
    .replace(/(^-|-$)/g, '');
```

### 2. Schema + FAQ generation block duplicated across both new pages

The BreadcrumbList + WebPage + Service + FAQPage JSON-LD construction, and the `faqSchema` conditional, are copy-pasted between `stages-thematiques.astro` and `ateliers-reguliers.astro`. They are also structurally identical to what `ContentPage.astro` produces.

This is the third (and fourth) duplicate of this pattern. A `src/utils/schema.ts` helper returning the standard page schema fragments would centralise it and make the `settings?.authorName` / `facebookUrl` references a single point of maintenance.

### 3. `audience` Keystatic field is stored but never displayed

`keystatic.config.ts:166` defines an `audience` select for each créneau (`adultes`, `ados-adultes`, `enfants`). It is read from the YAML but never rendered in `ateliers-reguliers.astro` — neither as a badge on the card nor as metadata in the detail section. The grouping logic uses the separate `group` field instead.

`audience` and `group` encode overlapping information (you can infer `revel-enfants` → audience `enfants`). Having both increases maintenance burden for Isabelle who has to keep them in sync.

**Options:**
- Remove `audience` and derive the display label from `group` in the template.
- Display `audience` as a visible badge on `AtelierCard`, which would add real value.

## Code Quality Improvements

### 4. Hover shadow in card components is a no-op

`StageCard.astro:28-30` and `AtelierCard.astro:32-34` both define:

```css
.stage-card { box-shadow: var(--shadow-ring), var(--shadow-whisper); }
.stage-card:hover { box-shadow: var(--shadow-ring), var(--shadow-whisper); }
```

The hover rule sets the same value as the default — no visual change occurs on hover. The cards are `<a>` elements, so hover feedback matters for usability. Either remove the dead rule or give it a distinct shadow (e.g., add `var(--shadow-lift)` on hover if that token exists, or slightly increase spread).

### 5. Hardcoded French section titles violate CLAUDE.md

CLAUDE.md states: *"All page content must be managed through Keystatic — never hardcode text in `.astro` page files."*

The following strings are hardcoded in templates:

| File | Hardcoded string |
|---|---|
| `stages-thematiques.astro:107` | `"Nos stages"` |
| `stages-thematiques.astro:145` | `"Prochaines dates :"` |
| `stages-thematiques.astro:164` | `"Questions fréquentes"` |
| `ateliers-reguliers.astro:116` | `"Nos créneaux"` |
| `ateliers-reguliers.astro:177` | `"Calendrier :"` |
| `ateliers-reguliers.astro:196` | `"Questions fréquentes"` |

Section titles (`"Nos stages"`, `"Nos créneaux"`, `"Questions fréquentes"`) should be optional Keystatic fields with sensible defaults, consistent with how `introductionTitle`, `actualitesTitle`, etc. are handled on the homepage. The label strings (`"Prochaines dates :"`, `"Calendrier :"`) are UI chrome and could be acceptable as shared i18n constants.

### 6. Redundant double guard on `stages.length > 0`

`stages-thematiques.astro:104` and `:126` independently check `stages.length > 0` for sections 3 and 4. They always appear and disappear together. A single fragment wrapper makes the dependency explicit and removes one evaluation:

```astro
{stages.length > 0 && (
  <Fragment>
    <!-- Section 3: Cards -->
    ...
    <!-- Section 4: Details -->
    ...
  </Fragment>
)}
```

### 7. `coverImageAlt: null` in both content files

Both `stages-thematiques/index.yaml:9` and `ateliers-reguliers/index.yaml` set `coverImageAlt: null`. Hero renders the cover image with no alt text — an accessibility and SEO gap. CLAUDE.md: *"All images must have descriptive alt text — never generic."*

**Fix:** Add a descriptive French alt string to both YAML files.

### 8. `dateModified` always generates today's build date

Both pages write `"dateModified": new Date().toISOString().split('T')[0]` in their WebPage schema. Every build marks every page as modified today regardless of actual content changes. For this static site this is a minor SEO inaccuracy (Google may discount the signal). An `updatedAt` Keystatic field per singleton, or accepting this as a known limitation with a comment, would be appropriate.

## Positive Observations

- **Consistent section pattern**: The intro → cards → details → FAQ → cross-links structure is applied uniformly across both pages and mirrors the homepage. The alternating accent/default background rhythm is maintained correctly.

- **Anchor nav is static/JS-free**: Using `<a href="#group-id">` anchor pills instead of client-side filters is the right call for a static site. It works without JavaScript, is SEO-transparent, and requires no state management.

- **Clean grouping logic in `ateliers-reguliers.astro:44-51`**: Computing `groupedCreneaux` by filtering against a typed `groups` array, then filtering out empty groups before rendering, cleanly prevents empty section headers without conditional clutter in the template.

- **`note || undefined` correctly suppresses empty prop**: `note={c.note || undefined}` avoids passing an empty string to `AtelierCard`, ensuring the conditional `{note && ...}` works correctly.

- **Keystatic field labels are editor-friendly**: Labels in French with inline examples (e.g., `'Prix (ex: 40 € ou 40 €–90 €)'`) will make the CMS usable for a non-technical editor without a manual.

- **`faqSchema` is null-safe**: The `faqItems.length > 0 ? {...} : null` pattern correctly suppresses the FAQPage schema when there are no FAQ items, avoiding invalid structured data.

- **`ateliers-reguliers-compare.astro` was cleanly patched**: Removing the `page.content` reference without deleting the comparison page was the pragmatic choice — the page remains available for historical reference without blocking the build.
