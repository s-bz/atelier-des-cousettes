## Architectural & Design Concerns

### 1. Orphan fields in Keystatic schema (`heroImage`, `heroImageAlt`)
**File:** `keystatic.config.ts` lines 63‚Äì66  
The `heroImage` and `heroImageAlt` fields remain in the schema and are populated in `index.yaml`, but `index.astro` no longer reads or renders them. These are dead fields that will confuse Isabelle in the CMS.

**Fix:** Remove `heroImage` / `heroImageAlt` from the schema and from `index.yaml`.

### 2. Section comment numbering is out of sync
**File:** `src/pages/index.astro`  
Comments read `<!-- 2. Introduction -->`, `<!-- 3. Pourquoi nous choisir -->`, `<!-- 5. Votre animatrice -->` ‚Äî section 4 was renumbered when introduction merged with cards but the comment wasn't updated. Minor but confusing during maintenance.

**Fix:** Renumber comments sequentially: 1 Hero, 2 Introduction + Cards, 3 Pourquoi nous choisir, 4 Animatrice, 5 Vid√©o, 6 Actualit√©s.

### 3. `ContactCTA` reads `siteSettings` on every instance
**File:** `src/components/ContactCTA.astro` line 9  
`ContactCTA` independently calls `reader.singletons.siteSettings.read()` every time it is rendered. On the homepage it is rendered **three times**, meaning three redundant singleton reads in the same request. Keystatic's `reader` is file-based and synchronous-ish, so in dev this is cheap, but it's wasteful and inconsistent with how other pages pass data down.

**Fix:** Accept `url` as a prop (which the homepage already has available via `settings?.contactFormUrl`) and pass it in. The internal fallback read can stay as a convenience for pages that don't pass it.

```astro
<!-- index.astro ‚Äî pass url explicitly -->
<ContactCTA label={homepage.ctaLabel || undefined} url={settings?.contactFormUrl || undefined} />
```

This avoids the triple read and makes data flow explicit.

## Code Quality Improvements

### 4. Repeated heading pattern across every section
**Files:** `index.astro`, `ValueProposition.astro`  
The h2 section heading pattern is repeated verbatim in every section:
```html
<h2 class="text-2xl md:text-3xl font-bold text-[var(--color-heading)] text-center mb-10"
    style="font-family: var(--font-heading);">
```
This appears 5 times across index.astro alone, plus once each in ValueProposition and AnimatriceSection. A `SectionHeading.astro` component (or a Tailwind `@apply` class like `.section-heading`) would DRY this up considerably. Not urgent given the project scale, but worth addressing before adding more sections.

### 5. `introduction2` field name is weak
**File:** `keystatic.config.ts` line 96  
`introduction2` is a code smell ‚Äî it implies a sequence that may grow (`introduction3`?). A better name would be `introductionSubtext` or the two fields could be replaced with a single `introductionText` that allows simple line breaks. The content is short enough that a `multiline` text field with a `\n\n` separator would work.

### 6. `[...homepage.valuePropositions]` spread is unnecessary noise
**File:** `src/pages/index.astro` line ~172  
The spread `[...homepage.valuePropositions]` exists purely to convert `readonly` to mutable for the `ValueProposition` component prop type. The better fix is to type the prop as `ReadonlyArray` in the component:

```ts
// ValueProposition.astro
interface Props {
  title: string;
  items: ReadonlyArray<{ title: string; description: string }>;
}
```

This removes the spread at the callsite and is more semantically correct ‚Äî the component never mutates the array.

### 7. `safeHref` fallback to `#` silently swallows bad data
**File:** `src/components/ServiceCard.astro` line 15  
```ts
const safeHref = href?.startsWith('/') ? href : '#';
```
A card with an empty or external href silently becomes a no-op `#` link. Since all hrefs come from Keystatic (trusted input), this guard is unnecessary. If a bad value ever gets in, the `#` behaviour is confusing ‚Äî a broken link is easier to debug than a card that goes nowhere. Remove the guard and trust the CMS.

### 8. `animatriceImage: null` in YAML with no fallback UI
**File:** `src/content/pages/homepage/index.yaml` line 38  
`animatriceImage` is `null`, so `AnimatriceSection` renders without an image ‚Äî the text column takes full width. This is currently handled via a conditional class:
```astro
<div class={image ? 'w-full md:w-3/5' : 'w-full max-w-2xl mx-auto text-center'}>
```
This works, but it means the animatrice section currently renders as a centred text block with no visual anchor. When a photo is eventually added, the layout will shift. Worth noting for Isabelle to add her photo.

## Security Findings

### 9. `set:html` with Markdoc-rendered content is safe ‚Äî confirm
**File:** `src/pages/index.astro` line ~224  
```astro
<Fragment set:html={actualitesHtml} />
```
`actualitesHtml` comes from `Markdoc.renderers.html()` on content authored in Keystatic. Markdoc sanitises its output by default (no raw HTML passthrough unless configured), so this is safe. The same pattern is used in blog posts site-wide. ‚úì

### 10. JSON-LD `replace(/<\//g, '<\\/')` is correct but undocumented
**File:** `src/pages/index.astro` line ~129  
The regex escaping of `</` is the correct standard technique to prevent `</script>` injection inside an inline `<script>` tag. It is not obvious to future maintainers. A one-line comment would help.

```ts
// Escape </script> sequences to prevent early tag termination in inline JSON-LD
]).replace(/<\//g, '<\\/')
```

## Type Safety

### 11. `settings` is read but never null-checked before destructuring phone
**File:** `src/pages/index.astro` lines 59‚Äì60  
```ts
const phone = settings?.phones?.[0]?.replace(/[.\s]/g, '');
const phoneIntl = phone ? `+33${phone.slice(1)}` : undefined;
```
Optional chaining is used correctly here. ‚úì However if `settings` is ever `null` (CMS misconfiguration), the JSON-LD schema silently emits `undefined` values for `email`, `telephone`, etc. These become `"undefined"` strings in JSON if not caught. The existing `if (!homepage)` guard pattern should be mirrored for `settings`:

```ts
if (!settings) throw new Error('Site settings not found');
```
