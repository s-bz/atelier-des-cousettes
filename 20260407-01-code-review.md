## Critical Issues

### 1. XSS via `set:html` on unvalidated Markdoc output â€” Medium severity

**Files:** `src/components/ContentPage.astro` (line 37), `src/pages/mes-creations.astro` (line 42), `src/pages/index.astro` (line 79)

```astro
<Fragment set:html={renderedContent} />
```

`Markdoc.renderers.html()` produces a raw HTML string that is injected verbatim into the DOM. For a CMS backed by a GitHub repo where the editor is the site owner this is low risk in practice, but it is worth noting:

- Any Markdoc custom tag that emits raw HTML (if added later) could introduce stored XSS.
- The `contentNode` type is typed as `any` in `ContentPage.astro` (line 11), removing compiler protection entirely.

**Recommendation:** Type `contentNode` properly. Consider using the Markdoc React/Astro renderer component instead of `renderers.html()` to keep output within a controlled component tree. At minimum, add a comment documenting the trust assumption.

### 3. `siteUrl` fallback vs. `Astro.site`

**File:** `src/components/ContentPage.astro` (line 19)


`Astro.site` is already configured in `astro.config.mjs` to `https://couture-tarn.fr`. The fallback string is redundant and should be removed â€” but more importantly, `mes-creations.astro` (line 33) hard-codes the URL directly in its structured data without using `Astro.site` at all:

## Architectural Concerns

### 4. Duplicated Markdoc transformation in every page

**Files:** `src/pages/index.astro` (lines 21â€“22), `src/pages/mes-creations.astro` (lines 10â€“11)

The three-line pattern:
```ts
const contentNode = page.content ? await page.content() : null;
const transformed = contentNode ? Markdoc.transform(contentNode.node) : null;
const renderedContent = transformed ? Markdoc.renderers.html(transformed) : '';
```

appears in both `index.astro` and `mes-creations.astro` in addition to being embedded inside `ContentPage.astro`. It is also structurally identical in every `ContentPage`-based page's frontmatter (the pages pass `contentNode` raw and `ContentPage` re-does the transform).

`ContentPage` correctly encapsulates the transform+render. The pages that _use_ `ContentPage` correctly offload everything to it. But `index.astro` and `mes-creations.astro` duplicate the logic by hand because they need custom layouts.

**Recommendation:** Extract a `renderMarkdoc(node)` utility function in `src/utils/markdoc.ts`. It is a 3-line helper but calling it in 4+ places vs. duplicating 3 lines is still the right architectural decision.

### 5. `contentNode: any` in ContentPage kills type safety

**File:** `src/components/ContentPage.astro` (line 11)

```ts
contentNode: any;
```

The Keystatic reader returns a typed async function. Once called, the result has a `.node` property typed by Markdoc. Using `any` here bypasses TypeScript entirely for this interface boundary.

**Recommendation:** Import or inline the correct type. The return type of `await page.content()` from a `fields.markdoc()` field is `{ node: Node }` from `@markdoc/markdoc`. Type it, or at minimum use `unknown` and narrow it.

### 6. Two cover images per page in the repository

**Directory:** `src/assets/images/covers/homepage/`

Each page's cover directory contains two files: a `coverImage.*` (the active one referenced in the CMS) and an old `homepage.jpg` / `la-couturiere.jpg` / etc. These originals are committed but never referenced. This doubles the image payload committed to the repo.

**Recommendation:** Delete the unreferenced originals or document explicitly why they are kept (e.g., as a backup). If they serve no purpose in production, clean them up.

### 7. Image glob scope in `Hero.astro` is global

**File:** `src/components/Hero.astro` (line 13)

```ts
const coverImages = import.meta.glob<{ default: ImageMetadata }>(
  '/src/assets/images/covers/**/*.{jpg,jpeg,png,webp}'
);
```

This eagerly includes every cover image across all pages in every page bundle. At build time Astro/Vite resolves this to a map of lazy imports, so actual module loading is deferred â€” but the full manifest is still inlined in every page that uses `Hero`. As the site grows, this becomes wasteful.

**Recommendation:** Pass the resolved image as a typed `ImageMetadata` prop rather than a path string. Each page imports its own image statically and passes it directly. This is the standard Astro pattern and eliminates the glob + runtime path-matching logic entirely.

## Code Quality Improvements

### 8. `legacy-peer-deps=true` in `.npmrc`

**File:** `.npmrc` (line 1)

This flag silences peer-dependency conflicts rather than resolving them. It is a common workaround for Keystatic's React version requirements but should be treated as technical debt with an associated comment noting which dependency causes the conflict and an issue/tracking item to remove it when upstream is fixed.

### 9. Inline `style` for font-family overrides Tailwind consistency

**Files:** `src/components/Hero.astro` (lines 37, 46)

```astro
style="font-family: var(--font-heading);"
```

This CSS variable is applied via inline `style` rather than a Tailwind utility. The rest of the codebase uses `var(--color-*)` variables in `class` attributes via arbitrary Tailwind values. Mixing inline styles and Tailwind utility classes for the same concern makes the styling less maintainable.

**Recommendation:** Register `--font-heading` as a Tailwind font-family utility (e.g., `font-heading`) so it can be used as `class="font-heading"` consistently.

### 10. Lightbox `alt` attribute is always empty at mount time

**File:** `src/pages/mes-creations.astro` (line 83)

```astro
<img id="lightbox-img" src="" alt="" class="..." />
```

The JavaScript opens the lightbox by setting `src` but never sets `alt`. A screen reader user who activates the lightbox will see no description of the image.

**Recommendation:** Add a `data-alt` attribute to the gallery buttons (alongside `data-src`) and set `lightboxImg.alt` in the click handler.

### 11. Missing `aria-label` on lightbox close button when using `Ă—` entity

**File:** `src/pages/mes-creations.astro` (line 82)


The `aria-label` is present â€” good. However, the `dialog` element itself has no `aria-labelledby` or `aria-label`. Screen readers announce the dialog but without a name. Consider adding `aria-labelledby` pointing to a visually hidden dialog title, or `aria-label` on the `<dialog>` element.

### 12. Hardcoded French text strings in components

**Files:** `src/components/Hero.astro` (line 27), `src/components/ContactCTA.astro` (line 7), `src/pages/mes-creations.astro` (lines 66, 76)

## Security Findings

### 15. Structured data content is not sanitised

**Files:** `src/components/ContentPage.astro` (line 24), `src/pages/index.astro` (line 27), `src/pages/mes-creations.astro` (line 29)

```astro
<script type="application/ld+json" set:html={JSON.stringify({...})} />
```

`JSON.stringify` does not escape the `</script>` sequence. If any CMS field value contains the literal string `</script>` (possible since the Keystatic editor accepts free text), it would break out of the `<script>` tag and could inject arbitrary HTML.

**Recommendation:** Use a utility that escapes the forward slash in `</script>` sequences: replace `</` with `<\/` after `JSON.stringify`. This is a standard practice for inline JSON-LD.

```ts
JSON.stringify(data).replace(/<\//g, '<\\/')
```
